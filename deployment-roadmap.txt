UniJobLink -> Real UCSI Deployment Roadmap
============================================

CONTEXT
-------
The project has been built and UI-polished as a local dev prototype (React/Vite frontend
talking to an Express/MySQL backend, both running on localhost). The goal is to take this
from "prototype for grading" to a real system that actual UCSI students, companies, and
academic supervisors would use with real personal data - not just a demo or portfolio
piece. This is a significant shift: it introduces real PII (names, UCSI IDs, resumes,
application history), real institutional stakeholders (UCSI IT, a project supervisor),
and real infrastructure costs/ownership questions that don't exist for a local dev
project. This plan sequences that transition - some steps are cheap now and expensive
later (schema migrations, file storage), and the institutional approval track has long,
unpredictable lead times that should start in parallel with the engineering work, not
after it.

Confirmed current stack (verified directly against the repo):
- Frontend: React 19 + Vite 8, react-router-dom 7, Tailwind CSS 4 (@tailwindcss/vite
  plugin), axios, socket.io-client, react-hot-toast, lucide-react.
- Backend: Express 5, Sequelize 6 ORM, mysql2 driver (MySQL), jsonwebtoken + bcryptjs,
  multer (file uploads), nodemailer (Gmail SMTP), socket.io server, node-cron.
- Database: MySQL, currently a local dev instance (DB_HOST=localhost, DB_USER=root,
  internship_db).
- Nothing exists yet for: git version control (no .git anywhere in the project), Docker,
  CI/CD, a migrations framework (schema is managed purely by
  sequelize.sync({ alter: true }) on every boot), or a backend/.gitignore.

A security audit was already completed earlier (5 vulnerabilities fixed: password-reset
token leak, TOCTOU race on offer-acceptance, stale-session deactivation, file-upload
extension/XSS risk, raw error-message disclosure). This plan does not repeat that work -
it covers what's still missing for a genuine production rollout.

New, codebase-verified findings that reshape priority order:
- The http://localhost:5000 origin is hardcoded in ~19 frontend files (including
  frontend/src/api/axios.js's axios baseURL, and both chat pages' SOCKET_URL constants)
  - there is zero import.meta.env usage anywhere in the frontend today. As it stands, the
  frontend literally cannot point at a deployed backend. This is a ~20-file refactor, not
  a small config tweak.
- backend/server.js hardcodes the same localhost origin allowlist twice (once for Express
  cors(), once for the Socket.IO server's CORS) - both need to become env-driven.
- backend/seed-admin.js creates a real admin login with a hardcoded plaintext password
  (Admin@12345) sitting in source control - must never run against prod as-is.
- No .gitignore exists at the repo root or in backend/ (only frontend/.gitignore exists),
  and backend/uploads/ already contains seeded dev files - sequencing matters here:
  gitignore must be written BEFORE the first git add, or purging already-committed
  uploads later requires a history rewrite.
- No express-rate-limit, helmet, or logging library in backend/package.json - confirms
  none of that hardening exists yet.
- Only 8 models total (User, Student, Company, Supervisor, JobPosting, Application,
  ChatRoom, Message) - small enough that migrating off sync({ alter: true }) to real
  migrations is a 1-2 day job, not a rewrite, and is far cheaper to do now (no real data
  to lose) than after real students are using it.


RECOMMENDED APPROACH - PHASED ROADMAP
--------------------------------------

PHASE 0 - This week (true blockers, no dependencies)

1. Git init + repo hygiene.
   git init at the project root (one repo for both backend/ and frontend/). Write a root
   .gitignore BEFORE the first git add, covering node_modules/, backend/.env,
   backend/uploads/* (use .gitkeep for empty dirs), frontend/dist, *.log. Verify with
   git status that .env and uploads/* are not staged before committing. Push to a
   PRIVATE GitHub repo. Every hosting platform in Phase 2 deploys by watching a connected
   git repo, so nothing else can start without this.

2. Fix the pervasive localhost:5000 hardcoding.
   Add frontend/.env + frontend/.env.example with VITE_API_URL; refactor
   frontend/src/api/axios.js's baseURL and the ~19 files building raw asset/socket URLs
   to use import.meta.env.VITE_API_URL instead. Make backend/server.js's CORS allowlist
   (both the Express and Socket.IO blocks) read from an ALLOWED_ORIGINS env var. Do this
   as one focused PR - mechanical but easy to miss a file. Nothing in Phase 2 can be
   verified end-to-end without this.

3. Migrate off sequelize.sync({ alter: true }) to sequelize-cli migrations, before any
   real student data exists (cheap now, expensive later - there's currently no live data
   to preserve). Add sequelize-cli, write one migration per existing model mirroring
   current attributes/associations, validate against a throwaway DB by diffing schemas,
   then recreate + re-seed the dev DB via migrations, and remove sync({ alter: true })
   from server.js entirely (replace with a plain sequelize.authenticate() boot check).
   From this point on, every schema change is a reviewed migration file, never
   auto-applied on boot.

PHASE 1 - Foundation (before any real user touches the system)

4. Move file uploads off local disk - must have.
   multer.diskStorage in backend/controllers/studentController.js and
   companyController.js writes resumes, profile images, and offer letters to local disk;
   most affordable Node hosts have ephemeral filesystems, so every upload vanishes on
   redeploy. Recommendation: Cloudflare R2 - S3-API-compatible (small storage-engine swap
   in multer), 10GB free with zero egress fees, decouples file durability from whichever
   compute host gets picked. Keep DB columns unchanged (still store a reference, just an
   object key instead of a relative path); serve private documents (resumes/offer
   letters) via short-lived presigned URLs rather than a public bucket.

5. Remaining auth/CORS hardening.
   Rate limiting on backend/routes/authRoutes.js (login/register/password-reset) via
   express-rate-limit. Fresh, high-entropy JWT_SECRET for prod (distinct from dev),
   least-privilege prod DB user (not root/blank password like dev). Rotate or remove the
   hardcoded seed-admin.js password before that script ever touches a prod DB. HTTPS
   itself is free from whichever host is chosen in Phase 2, but depends on Phase 0 item 2
   being done first.

PHASE 2 - Hosting & infrastructure decision

Three pieces: frontend (static), backend (Node/Express + Socket.IO), database (MySQL).

- First choice: ask UCSI (FCI/IT) for sponsored hosting - pursue in parallel with Phase 3
  starting now. Solves cost and the data-residency/PDPA conversation at once if UCSI
  provides the infrastructure.
- Self-funded fallback: Railway (backend + MySQL bundled, git-push deploys, ~$5-10/mo
  after trial credit) + Vercel/Netlify for the static frontend (free, no cold starts).
- Zero-budget fallback: Render free tier for both frontend and backend + an external
  managed MySQL free tier - but Render's free web service sleeps after ~15min idle
  (10-50s cold start on wake), which is a bad look for a real portal companies use under
  deadline pressure. Treat as pilot/beta-only, not full rollout.
- Whichever host is chosen, get a real domain or ask UCSI IT about a subdomain (e.g.
  unijoblink.ucsi.edu.my) - all shortlisted hosts auto-provision HTTPS for custom
  domains.

PHASE 3 - Institutional / non-engineering track (start now, run in parallel)

1. Supervisor sign-off that the project is shifting from "graded prototype" to "real
   system holding real student PII" - cheapest, fastest conversation, have it first.
2. UCSI IT / InfoSec review - required for real student PII and for any UCSI-branded
   subdomain/sponsored server ask. Likely slow (weeks) - start early; may also settle the
   Phase 2 hosting question.
3. PDPA 2010 (Malaysia) awareness, not legal advice: plain-language data-collection
   notice at registration, a manual way for a user to request deletion/export, a stated
   post-graduation retention policy, and a check that role-based access (the existing
   authenticateToken/authorizeRoles middleware) actually prevents e.g. a supervisor from
   browsing unrelated students' resumes. Raise formally with UCSI's compliance/legal
   office once IT is looped in.
4. Long-term ownership after graduation - decide now whether UCSI adopts this as an
   IT/faculty-owned system, or it's explicitly scoped as a time-boxed pilot with an
   announced decommission/data-purge date. "The student maintains it forever as a
   volunteer" isn't a sustainable answer for a system holding real people's data.

PHASE 4 - Remaining hardening before go-live

- Automated DB backups (must have) - check if the managed MySQL provider includes them;
  if self-managed, add a scheduled mysqldump job (the project already uses node-cron for
  the offer-expiry job, so this is a small addition) into the same object storage bucket
  as uploads, with a retention policy.
- Basic /health route + free uptime monitoring (must have, cheap) - a ~10-line route
  checking sequelize.authenticate(), watched by UptimeRobot or similar.
- Structured logging, Dockerfile, full CI pipeline - nice to have, likely skip for now.
  The test script is currently a stub with no tests, so a CI pipeline has nothing
  meaningful to run yet; Render/Railway/Vercel's native auto-deploy-on-push already
  covers most of the practical benefit for a solo maintainer.


MUST-HAVE VS NICE-TO-HAVE
--------------------------
Must have before real UCSI data:
  - Git init + correct .gitignore sequencing
  - Fix ~20-file localhost:5000 hardcoding
  - Env-driven CORS (Express + Socket.IO)
  - Migrate off sync({alter:true}) to migrations
  - Move uploads off local disk (Cloudflare R2 or equivalent)
  - Rate limiting on auth routes
  - Fresh prod JWT_SECRET + prod DB creds, distinct from dev
  - Rotate/remove hardcoded seed-admin password before prod use
  - Automated DB backups
  - Basic /health + uptime monitoring
  - Supervisor sign-off, UCSI IT review, PDPA-aware practices, post-grad ownership
    decision

Nice to have / can wait:
  - Dockerfile
  - Full CI pipeline
  - Structured logging (Winston/Pino)
  - Branch protection / PR review process

Critical files (Phase 0-1 engineering work):
  - backend/server.js (CORS allowlists, sync({alter:true}) removal, /health route)
  - frontend/src/api/axios.js (hardcoded baseURL)
  - backend/controllers/studentController.js, companyController.js (multer storage)
  - backend/models/index.js, backend/config/database.js (migrations source of truth)
  - backend/.env.example, new frontend/.env.example
  - backend/seed-admin.js (hardcoded password)


VERIFICATION
------------
1. Git: git status shows a clean tree with .env/uploads/* untracked; git log shows the
   initial commit; repo visible on GitHub as private.
2. Env refactor: grep the frontend for the literal string "localhost:5000" - zero
   matches once done. Start both dev servers with VITE_API_URL pointed at a different
   port and confirm the app still works unchanged.
3. Migrations: sequelize-cli db:migrate against a fresh empty DB produces a schema
   identical (via mysqldump --no-data diff) to the current dev DB's schema; app boots and
   functions normally afterward with sync() removed from server.js.
4. File storage: upload a resume/profile image/offer letter in each relevant flow,
   confirm it lands in R2 (or chosen store) and is retrievable; restart the backend
   process and confirm the file still resolves (proving it survived a restart, unlike
   local disk on an ephemeral host).
5. End-to-end deploy: once Phase 2's host is chosen, do a full manual smoke test against
   the deployed URL - register each of the 4 roles, log in, upload a resume, apply to a
   job, accept/reject an offer, send a chat message - before inviting any real UCSI user.
