================================================================================
 UNIJOBLINK (UCSI INTERNSHIP PORTAL) — PROJECT DOCUMENTATION & CHANGE LOG
================================================================================

Project:      UniJobLink — UCSI University's official internship management and
              placement platform.
Users:        Students, Companies (HR partners), Academic Supervisors, Admins.
Tech stack:   Frontend — React 19 (Vite), plain CSS (index.css) + inline styles,
                         no Tailwind component classes (Tailwind is only used for
                         a handful of layout visibility toggles).
              Backend  — Node.js / Express, Sequelize ORM, MySQL, Socket.IO
                         (real-time chat/notifications).
Design system: Desktop uses a deliberately sharp, 0px-radius "editorial minimal"
              style. Mobile (<=768px) uses a warmer, rounded, app-like style.
              Golden rule followed throughout this project: desktop layout must
              never change unless explicitly requested — all mobile-specific
              layout changes live inside "@media (max-width: 768px)" in
              frontend/src/index.css.

Purpose of this document: a running record of what was built, what was broken,
what was changed, and why — organised into phases in the order the work actually
happened, for project documentation / FYP write-up purposes.

 PHASE 2 — MOBILE CLICK-THROUGH FIXES + IN-APP PDF VIEWER  (v1)
--------------------------------------------------------------------------------
Reported bugs (mobile only):
  1. Company Portal: "Archive Job" and "Delete Job" buttons in the job-edit
     Danger Zone did nothing when tapped.
  2. "View Resume PDF" did nothing when tapped, for every applicant, across the
     Company Applicant Pipeline.
  3. Feature request: tapping "View Resume" should open an in-app PDF
     viewer/popup letting the user view or download the PDF, instead of the
     (broken) old behaviour.

Root causes found:
  - Danger Zone buttons: both handlers started with a bare
    "if (!confirm(...)) return;" — window.confirm() is a native browser dialog
    known to be unreliable/silently suppressed in various mobile WebViews/in-app
    browsers.
  - Resume/offer links: the exact same "unclickable on mobile" stacking-context
    bug already fixed once for .mobile-app-resume-link, but never ported to the
    Applicant Pipeline's own resume/offer-letter links.

Changes made:
  - backend/controllers/companyController.js — added hardDeleteJob (validates
    ownership, blocks deletion if the job has any Application history, otherwise
    destroys it). backend/routes/companyRoutes.js — added the corresponding
    DELETE /company/jobs/:id/permanent route. (The pre-existing DELETE
    /company/jobs/:id route was already a soft-delete/archive, not a true
    delete — confirmed via code inspection before adding the new route.)
  - frontend/src/pages/company/CompanyManageJobs.jsx — Danger Zone's
    handleDelete/handleHardDelete rewritten to use the app's own ConfirmModal
    (via a small confirmAction state) instead of window.confirm().
  - frontend/src/index.css — added the proven z-index/pointer-events fix to
    .mobile-form-actions button and to .applicant-card-actions a.
  - NEW frontend/src/components/PdfViewerModal.jsx and
    frontend/src/hooks/usePdfViewer.js — a reusable "view this PDF" hook +
    modal: on mobile, opens an in-app modal; on desktop, behaviour is
    unchanged (window.open in a new tab). First version used a plain
    <iframe src="..."> to display the PDF, built on the existing
    .layer-overlay/.layer-panel modal shell.
  - Rolled out to every resume/offer-letter link app-wide (11 links across 6
    files): ApplicantPipelineParts.jsx (ApplicantCard + ClosedApplicantCard),
    StudentProfile.jsx, StudentApplications.jsx, StudentProfileModal.jsx,
    SupervisorStudentDetail.jsx, StudentDashboard.jsx. Two other files
    (CompanyATS.jsx, ApplicantPipelinePanel.jsx) were checked and confirmed to
    only pass data through as props — no direct link to convert.

 PHASE 3 — PDF VIEWER BUG-FIX ROUND  (v2 — the working version)
--------------------------------------------------------------------------------
User report after live mobile testing of Phase 2's PDF viewer: tapping "View
Resume" immediately prompted a native file download instead of showing anything;
the popup, when it did appear, looked like a full-page takeover instead of a
small popout box; and the underlying "View Resume" button was visible showing
through/behind the popup.

Investigation & root causes:
  - Mobile browsers (Chrome for Android, many in-app WebViews) generally cannot
    render a PDF embedded in a plain <iframe> the way desktop browsers can —
    when they can't, the common fallback is to auto-download the file instead.
    This was the real cause of both the "prompts a download" and "viewer doesn't
    work" symptoms — one root cause, two symptoms.
  - The popup reused .layer-overlay/.layer-panel, whose shared mobile CSS forces
    a 100vw/100dvh full-screen takeover (correct for other modals that use that
    shell, wrong for this one).
  - Live testing in a real mobile-width browser session found a second, more
    subtle bug: the new popup's z-index (99999) exactly tied with the z-index
    already given to the resume link itself in Phase 2's CSS fix. On a tie,
    later DOM order wins the paint order — and the underlying button happened
    to be later in the DOM, so it visually rendered on top of the popup.

Changes made (frontend/src/components/PdfViewerModal.jsx rewritten):
  - Added the "react-pdf" npm package (wraps pdfjs-dist) so the PDF is rendered
    onto a <canvas> by the app itself, independent of whatever PDF support the
    device's browser/WebView does or doesn't have.
  - Replaced .layer-overlay/.layer-panel with new dedicated classes
    (.pdf-viewer-overlay / .pdf-viewer-panel, added to index.css inside the
    mobile media query) sized as a genuine small centered popout box (~480px
    wide, ~82vh tall, capped height) instead of a full-screen takeover.
  - Download button switched from a plain <a href download> (silently ignored
    cross-origin on some browsers) to a fetch -> blob -> objectURL ->
    programmatic-click flow, which reliably forces a real save regardless of
    origin.
  - Bumped the popup's z-index to 999999 so it can never tie with any other
    element's 99999 in this codebase again.
  - Verified live in a real headless-browser session at both mobile and desktop
    viewport widths, with a real (non-blank) resume PDF, including confirming
    the Download button triggers an actual browser file-download event and
    that desktop's existing window.open-in-new-tab behaviour is completely
    unchanged.

 PHASE 6 — THEME TOGGLE REDESIGN (APPLE/iOS-STYLE SWITCH)
--------------------------------------------------------------------------------
Report: the dark/light mode control was a bare Sun/Moon icon button with no
track or pill shape — looked "weird" on both mobile and desktop. Requested an
Apple-style sliding switch instead (pill track, sun/moon at each end, a knob
that slides left/right).

This exact bare-icon-button snippet was duplicated 8 times across the app (2
occurrences — desktop sidebar header + mobile top header — in each of
CompanyLayout.jsx, StudentLayout.jsx, SupervisorLayout.jsx, AdminLayout.jsx).

Changes made:
  - NEW frontend/src/components/ThemeToggle.jsx — a single reusable component
    (pulls toggle/isDark/c from the existing useTheme() hook itself) rendering
    a pill-shaped track with a Sun icon fixed at the left end, a Moon icon fixed
    at the right end, and a circular knob that slides between them via a CSS
    transform transition. The whole track is the click target.
  - All 8 call sites across the 4 layout files swapped to <ThemeToggle />; the
    now-unused Sun/Moon lucide-react imports and isDark/toggle destructuring
    were removed from those files.
  - Bug found during live verification: a pre-existing global mobile CSS rule
    ("a, button, [role='button'] { min-height: 44px; min-width: 44px; }" —
    a Material-Design touch-target-size rule applying to every button on
    mobile) was overriding the switch's intended small size via min-height,
    blowing it up into an oversized 44px-tall blob. Fixed by adding a scoped
    ".theme-toggle { min-height: 28px !important; min-width: 50px !important; }"
    override — a deliberate, explained exception, since a slider switch is
    conventionally smaller than a tap target (iOS's own switch is ~31px tall).
  - Verified live at both breakpoints; confirmed identical rendering across
    Student and Supervisor portals (Company/Admin share the same component so
    are covered by construction).
